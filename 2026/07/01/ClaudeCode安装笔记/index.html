Claude Code 安装与使用入门笔记

前言

Claude Code 是一个运行在命令行里的 AI Agent 工具。它不只是“会聊天的 AI”，而是可以在指定项目目录中读取文件、执行命令、安装工具、调用 MCP、使用 Skills，并围绕一个真实工作目录完成连续任务。

本文整理一套 Windows 环境下的 Claude Code 安装与基础使用流程，内容包括：

  • 安装 Claude Code；
  • 配置第三方模型接口；
  • 安装 Playwright MCP，让 Claude Code 控制浏览器；
  • 理解工作目录、权限模式、上下文和历史会话；
  • 安装 Skills，并用能力包完成更复杂的自动化任务。

一、先理解几个核心概念

1. Claude Code 不只适合程序员

Claude Code 最常见的使用场景确实是写代码、改代码和排查工程问题，但它本质上更接近一个通用 Agent 运行环境。

在实际使用中，它可以做的事情包括：

  • 读取和修改本地文件；
  • 执行终端命令；
  • 安装依赖和工具；
  • 调用浏览器、文档、剪辑等外部能力；
  • 通过 MCP 连接第三方工具；
  • 通过 Skills 复用固定工作流；
  • 把阶段性结果写入 Markdown、代码文件或其他项目资产。

所以不要只把它理解成“写代码工具”，它更像是一个能直接操作本地工作区的 AI 工作台。

2. Claude Code 和 Claude 订阅不是一回事

Claude Code 可以理解成一层 Harness，也就是负责组织上下文、调度工具、管理权限和调用模型的运行框架。

真正负责推理和生成内容的是底层模型。这个模型可以是 Anthropic 官方模型，也可以是兼容 Anthropic 接口格式的第三方模型服务。

也就是说，Claude Code 本体负责“怎么干活”，模型服务负责“怎么思考”。

3. MCP 和 Skills 的区别

MCP 更像是工具连接协议，用来让 Claude Code 接入浏览器、文件系统、数据库、设计工具等外部能力。

Skills 更像是能力包，通常会把工具接入方式、提示词、执行流程和注意事项封装在一起。相比单纯装一个工具，Skills 更强调“按某种方法把任务做完”。

4. 使用 CCSwitch 接入 GPT-5.5

如果不想反复手动修改 .claude.json，也可以使用 CCSwitch 管理模型配置。CCSwitch 的作用是统一管理 Claude Code、Codex、Gemini CLI 等 AI 编程工具的 Provider 配置，适合在多个模型服务之间切换。

接入 GPT-5.5 时，核心思路是：在 CCSwitch 中新增一个 Provider，然后把 Claude Code 切换到这个 Provider。具体字段以你使用的服务商为准，通常需要配置：

  • Provider Name：自定义名称，例如 GPT-5.5；
  • Base URL：服务商提供的接口地址；
  • API Key：服务商提供的密钥；
  • Model：模型名，例如 gpt-5.5；
  • Target / Tool：选择 Claude Code，或启用 CCSwitch 对 Claude Code 的配置接管；
  • Protocol / Format：根据服务商选择 OpenAI-compatible、Anthropic-compatible，或使用 CCSwitch 的本地代理做协议适配。

保存 Provider 后，在 CCSwitch 中切换到这个配置，再重新启动 Claude Code：

如果配置正确，Claude Code 会通过 CCSwitch 指向 GPT-5.5 所在的模型服务。这里需要注意：gpt-5.5 必须是你的服务商实际支持的模型名，如果接口返回 model_not_found，优先检查模型名、Base URL 和 Key 是否匹配。

另外，CCSwitch 只能管理和切换配置。它不会把 ChatGPT 网页会员自动变成 Claude Code 可用的 API Key。如果你使用的是 API 服务或中转服务，就按服务商给出的 Base URL / Key 配置；如果你使用的是 Codex + ChatGPT 账号登录，那更适合在 Codex 侧使用，而不是把 ChatGPT 登录信息写进 Claude Code。

六、为什么 Agent 执行任务会比较慢

普通聊天工具通常是“一次输入、一次输出”。

Claude Code 这类 Agent 工具不是这样。它完成一个任务时可能会经历多轮循环：

  1. 调用模型理解任务；
  2. 读取项目文件；
  3. 再调用模型分析；
  4. 执行终端命令；
  5. 检查命令输出；
  6. 遇到错误后重新判断；
  7. 修改文件；
  8. 再次验证结果。

所以一个看起来简单的任务，背后可能包含几十次模型调用和工具调用。

这也是 Agent 比普通聊天慢，但能完成真实工程任务的原因。

3. 理解 Claude Code 的界面输出

执行任务时，Claude Code 界面中会出现不同类型的信息：

  • 你的输入：你下达的任务；
  • 模型回复：Claude Code 对任务的解释和执行计划；
  • 文件读取：它正在查看哪些文件；
  • 工具调用：它正在执行哪些命令或能力；
  • 错误提示：工具执行失败、权限不足或依赖缺失；
  • 底部状态：当前工作耗时、token 使用量等信息。

不需要一开始全部看懂，只要知道它并不是卡死，而是在持续执行任务。

如果它请求执行某个命令，需要你确认是否允许。确认前建议看一下命令内容，尤其是删除文件、移动文件、安装依赖这类操作。

十、权限模式与审批

默认情况下，Claude Code 执行很多操作前都会请求批准，例如：

  • 执行命令；
  • 安装依赖；
  • 修改文件；
  • 调用外部 MCP；
  • 访问浏览器或网络。

这种设计是为了安全，但频繁审批会影响连续任务效率。

如果你清楚当前任务和工作目录的风险，可以根据情况调整权限模式。越宽松的权限模式，自动化程度越高，但风险也越大。

建议原则：

  • 不熟悉的项目，先保持默认审批；
  • 不确定命令含义时，不要直接同意；
  • 不要在包含敏感文件的目录里随意放宽权限；
  • 使用来源不明的 MCP 或 Skills 时保持谨慎；
  • 只有在自己明确知道任务边界时，再考虑更激进的权限配置。

十四、总结

到这里，Claude Code 的核心入门流程基本已经完整：

  • 本地安装并验证 claude 命令；
  • 配置模型服务和 API Key；
  • 在项目目录中启动 Claude Code；
  • 使用 Playwright MCP 控制浏览器；
  • 通过 /resume、/context、/compact、/clear 管理会话和上下文；
  • 使用 Skills 扩展更复杂的任务能力。

Claude Code 真正有价值的地方，不只是回答问题，而是能在一个具体目录中连续执行任务。日常使用时，建议把重要规则、阶段总结和产出都沉淀到文件里，让文件成为长期记忆，让上下文只承担短期任务沟通。
