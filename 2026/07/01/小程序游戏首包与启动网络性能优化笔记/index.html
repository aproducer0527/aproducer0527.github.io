抖音小游戏首包与启动网络性能优化笔记

项目背景

某 Unity WebGL 抖音小游戏在抖音开发者工具性能检测中出现首包体积与网络性能问题：

  • 首包大小超过平台限制，目标需控制在 30MB 以内。
  • FP 阶段存在 tt.request 平均耗时偏高问题。
  • 构建产物主要体积集中在 webgl.data，资源治理优先级高于代码体积优化。

优化目标

  • 降低 Unity WebGL 首包体积，满足抖音小游戏首包限制。
  • 减少首屏阶段非必要 TTSDK 请求。
  • 保持广告、侧边栏奖励等商业化功能可用，但改为按需触发。
  • 建立可复用的资源扫描、导入配置与构建验证流程。

问题定位

通过 Unity Build Report 与构建产物分析，确认主要问题包括：

  • 大体积字体源文件被直接打入首包。
  • TMP 示例资源位于 Resources 目录，被 Unity 自动收集进包体。
  • 可选 UI prefab、图片资源放在 Resources 下，导致未使用资源进入 webgl.data。
  • 部分纹理、音频资源导入参数不适合小游戏首包场景。
  • 启动期存在 TTSDK 初始化、侧边栏可用性检查、加载场景自动插屏等非必要网络调用。

核心优化方案

1. 首包资源治理

  • 基于 Build Report 分析 webgl.data 内资源占比，优先处理高体积资源。
  • 移除或禁用无关示例资源，例如 TextMeshPro 示例 Resources。
  • 将非首屏、非必需 UI 资源从 Resources 目录迁移到可选目录，避免 Unity 自动打包。
  • 对侧边栏奖励弹窗改为运行时动态构建 UI，减少 prefab 与图片依赖。
  • 关闭源字体 includeFontData，仅保留实际使用的 TMP SDF 字体资产。

2. 纹理与音频导入优化

  • 对大图资源统一设置平台压缩参数。
  • 控制最大贴图尺寸，避免 2D UI 或特效图以原始尺寸进入包体。
  • 启用适合 WebGL/小游戏场景的纹理压缩策略。
  • 对大体积音频降低压缩质量。
  • 使用 AudioImporter.SampleSettings 替代过期 API，关闭非必要 preloadAudioData。
  • 保证优化脚本兼容 Unity 新版本，避免 obsolete API 导致编译失败。

3. WebGL 构建配置优化

  • 关闭 Development Build、Profiling、Debug Symbols 等调试选项。
  • 启用代码裁剪与 WebGL 压缩相关配置。
  • 调整小游戏平台构建参数，降低首包冗余内容。
  • 保留可复用 Editor 工具，用于一键应用资源导入策略和构建配置校验。

4. 启动期网络请求优化

  • 移除启动阶段的 TTSDK 主动初始化。
  • 移除侧边栏控制器与插屏广告控制器的启动自动创建逻辑。
  • 将 CheckScene、NavigateToScene、CreateInterstitialAd 等 TTSDK 调用改为用户行为触发。
  • 关闭加载场景自动插屏，避免首屏阶段触发广告相关请求。
  • 保留通关、重开、退出、下一关等业务场景下的广告触发能力。
  • 对侧边栏可用性结果增加缓存，避免重复请求。

验证方式

  • 使用 Build Report 对比资源打包明细。
  • 分析 zip 构建产物中 webgl.data、webgl.wasm 占比。
  • 使用静态扫描确认：
    • 无启动期侧边栏可用性检查。
    • 无加载场景自动插屏。
    • 无侧边栏 prefab 通过 Resources.Load 首屏加载。
    • 示例资源不再位于 Resources 目录。
    • 字体源文件不再携带字体数据进入包体。
  • 使用 dotnet build Assembly-CSharp.csproj 验证 C# 编译无错误。
  • 最终结果需以抖音开发者工具重新构建后的性能检测为准。

阶段性收益

  • 首包从约 45MB 优化到约 36MB。
  • 进一步移除字体源文件、TMP 示例资源、可选侧边栏 UI 资源后，预计可继续减少约 15MB。
  • 优化后首包预计可低于 30MB 平台限制。
  • 启动期 tt.request 从自动触发调整为按需触发，降低 FP 阶段网络耗时风险。

可复用经验

  • Unity 小游戏首包优化应优先分析 webgl.data，而不是只关注脚本体积。
  • Resources 目录是首包膨胀高风险点，应严格控制进入条件。
  • 字体源文件、TMP 示例资源、未使用 prefab 是常见隐性体积来源。
  • TTSDK、广告、侧边栏等平台能力应避免在 Awake、RuntimeInitializeOnLoadMethod 中主动请求。
  • 商业化能力可以保留，但应通过 Lazy Initialization 与用户行为触发降低首屏压力。
  • 资源优化必须配合构建报告、静态扫描和实际平台检测闭环验证。
